CWI Interpretation
1. What CWI Means in This Project

CWI stands for Combinatorial Workflow Intelligence.

In this project, CWI refers to an analytical frame that interprets organizational workflows as combinations of actors, documents, systems, roles, events, constraints, and outcomes.

CWI is not used here as a standardized academic term. It is used as a project-level framework for analyzing workflow complexity, onboarding failure, and knowledge-transfer breakdown.

2. Why CWI Fits This Case

The remote ERP onboarding case was not a simple training problem.

It involved many interacting elements.

New employee
Senior consultants
customers
developers
ERP systems
external platforms
training documents
meeting notes
upload templates
self-study files
system screens
videos
credentials
project deadlines
customer questions
production and inventory artifacts

CWI is useful because the failure emerged from combinations of these elements rather than from one single cause.

3. Event-Based Interpretation

Each onboarding artifact can be treated as an event signal.

Examples:

meeting summary created
training video watched
customer document received
upload template reviewed
keyword file created
presentation deck drafted
comparison file analyzed
BOM template examined
work order concept introduced
feedback received or missing

These events can be organized into a synthetic onboarding event log.

4. Role-Based Interpretation

The new employee’s role shifted across several modes.

learner
observer
note-taker
translator
researcher
presentation preparer
data checker
ERP concept learner
project support person
self-onboarding manager

CWI helps identify when role expectations become too broad or unclear.

5. Artifact-Based Interpretation

The case shows that artifacts can either support learning or overload learning.

A well-designed artifact system separates materials by status and purpose.

A fragmented artifact system forces the learner to classify everything alone.

CWI treats artifacts as part of the workflow, not as passive files.

6. Knowledge-Transfer Interpretation

Knowledge transfer failed when documents, meetings, videos, and assignments were not connected by a staged learning pathway.

In this case, knowledge did move.

However, it moved through fragmented artifacts rather than through a structured curriculum.

This means that knowledge transfer existed but was not sufficiently governed.

7. Operational Intelligence Interpretation

The case also shows how onboarding can be analyzed as an operational process.

A company can monitor onboarding through:

number of artifacts assigned
number of domains introduced
number of unresolved questions
number of outputs expected
number of feedback events
number of role changes
number of customer contexts introduced
time between assignment and review

These indicators can help detect onboarding overload before failure occurs.

8. Connection to SAP-Style CWI

The same logic can be extended to SAP or ERP process analysis.

In ERP systems, event logs can show how business processes actually move.

In onboarding systems, artifact logs can show how learning and role expectations actually move.

SAP-style CWI would combine:

event logs
workflow maps
role matrices
approval structures
knowledge-transfer gaps
bottleneck detection
exception taxonomy
operational risk reporting

This case can therefore serve as a human-centered origin case for SAP-style CWI.

9. Key Finding

CWI reframes the case from a personal failure story into a workflow-system problem.

The question is not simply whether a new employee worked hard enough.

The question is whether the onboarding system gave the employee the right sequence of knowledge, role clarity, artifacts, feedback, and operational boundaries.

In this case, the available materials suggest that these elements were not properly aligned.
