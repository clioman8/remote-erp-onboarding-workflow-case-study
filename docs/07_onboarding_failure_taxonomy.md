Onboarding Failure Taxonomy
1. Purpose

This document defines the major failure categories observed in the remote ERP onboarding case.

The purpose is to convert personal experience and fragmented artifacts into a reusable analytical taxonomy.

This taxonomy can later be used to code synthetic event logs, case notes, interview data, and onboarding artifacts.

2. Failure Type 1: Live Project Exposure Before Scaffolding

This failure occurs when a new employee is exposed to live customer project complexity before they understand the core product, process structure, customer context, and role expectations.

Typical signals include:

customer meeting notes appear before basic training
unresolved implementation questions become learning material
customer-specific exceptions appear before general principles
live project documents replace staged exercises
3. Failure Type 2: Artifact Fragmentation

This failure occurs when documents are not separated by purpose, domain, status, or learning sequence.

Typical signals include:

training files mixed with live customer files
drafts mixed with final documents
meeting notes mixed with upload templates
credential-related files mixed with learning artifacts
customer-specific data mixed with general references
4. Failure Type 3: Role Ambiguity

This failure occurs when the new employee’s role is unclear.

The employee may be treated as learner, observer, note-taker, translator, analyst, presenter, data checker, or consultant assistant without explicit boundaries.

Typical signals include:

unclear deliverables
unclear responsibility level
unclear authority
unclear review process
unclear ownership of output
5. Failure Type 4: Domain Overload

This failure occurs when too many business domains are introduced at the same time.

Observed domains include:

sales order
return process
e-commerce integration
WMS and 3PL
data reconciliation
manufacturing
BOM
routing
work order
MRP
costing
accounting
intercompany
access control
6. Failure Type 5: Knowledge Status Ambiguity

This failure occurs when the learner cannot tell whether a document is official, provisional, outdated, customer-specific, internal, or experimental.

Typical questions include:

Is this the official process?
Is this only a meeting note?
Is this customer-specific?
Is this a draft?
Is this something I must execute?
Is this safe to use?
Is this still valid?
7. Failure Type 6: Feedback Loop Weakness

This failure occurs when work is assigned or materials are provided, but understanding is not systematically checked.

Typical missing elements include:

weekly learning goals
check-in questions
progress review
misunderstanding correction
feedback on notes
feedback on presentations
role-specific next steps
8. Failure Type 7: Self-Onboarding Burden

This failure occurs when the employee must build their own learning system because the official onboarding structure is insufficient.

Typical signals include:

personal keyword files
personal study sheets
self-made concept maps
reconstructed process notes
privately organized customer files
independent terminology tracking
9. Failure Type 8: Security and Access Boundary Risk

This failure occurs when account, credential, system access, customer data, or sensitive references are mixed with learning files or project artifacts.

The issue is not whether misuse occurred.

The issue is that onboarding should clearly separate:

training materials
customer data
access credentials
permission rules
system roles
offboarding or access revocation procedures
10. Failure Type 9: Video-Based Knowledge Compression

This failure occurs when long screen-sharing videos are used as major knowledge-transfer tools without modular structure.

Typical missing supports include:

table of contents
timestamp index
practice checklist
concept glossary
follow-up exercise
review questions
short module separation
11. Failure Type 10: Master Data Governance Overload

This failure occurs when new employees are exposed to high-risk ERP master data issues too early.

Examples include:

Item creation
Costing Method
Unit Type
External ID
BOM
BOM Revision
Routing
Lot/Serial
Planning rules
master data import
irreversible or difficult-to-change settings
12. Taxonomy Summary

The case shows that remote ERP onboarding can fail even when many documents, meetings, videos, and training materials exist.

The problem is not simply absence of materials.

The problem is misalignment among learning stage, artifact status, workflow sequence, role expectation, feedback, and operational responsibility.
